# OL-ECOM-009: eBay Feed `COMPLETED_WITH_ERROR` Result File

Status: accepted

Decision: `accept_online_case`

Offline reproduction: `not_possible`

Idea-bank route: forbidden

## Platform And Surface

- Platform: eBay Sell Feed API
- API surface: Seller Hub feed upload, order reports, order acknowledgement
- Endpoints/concepts: `createTask`, `uploadFile`, `getTask`,
  `getResultFile`, `FeedStatusEnum`

## Drift Category

Async partial-success and per-record error visibility drift.

## Checked Situation

eBay's Seller Hub Feed flow is asynchronous. Official docs say clients create a
task, upload the file, then poll task status until `COMPLETED` or
`COMPLETED_WITH_ERROR`. The same docs state that the task response can expose
success and failed record counts through `uploadSummary`, and that failed
requests require downloading the result file because the file contains the
issues or errors. The `FeedStatusEnum` reference defines
`COMPLETED_WITH_ERROR` as a completed task where errors are indicated in the
report.

The silent integration risk is that an old client treats any terminal
"completed" task as a successful feed and never downloads or parses the result
file. The API calls can all succeed while some listings, fulfillments, or order
acknowledgements failed at record level, leaving local ERP state ahead of eBay.

## Why Offline Reproduction Is Not Possible

The proof requires eBay seller credentials, marketplace-specific feed templates,
live task creation, and the result file generated by eBay's service. A local
fixture can demonstrate the parser risk, but not the live platform behavior.
